项目背景

解决方案

项目描述
先介绍项目背景，再说一开始是怎么做的，但是遇到了什么困难、挑战，尝试了什么新的方法，结论是什么（数值型评估指标），一定要突出项目"迭代"的过程。
这个项目的主要目标是开发一个智能旅行规划助手。公司希望利用大模型技术给用户提供个性化的行程规划服务，我负责的是整个多Agent系统的架构设计和核心模块实现。之所以要做这个系统，是因为传统的旅行规划方式效率很低、体验也不好。很多用户想规划一次旅行，得在不同平台上查景点、看攻略、订酒店、查交通，至少要花2-3个小时，而且系统完全不记得用户上次去哪玩过、喜欢住什么酒店这些偏好信息，每次都要重新输入，非常麻烦。

最初尝试用简单的Agent架构来解决这个问题。我们基于AgentScope框架实现了意图识别和行程规划两个基础Agent，采用了串行调度的方式。具体来说，就是IntentionAgent先识别用户意图（是想查询、规划还是咨询政策），然后调用对应的ItineraryPlanningAgent生成行程。这个初版系统能够处理"我想去杭州玩三天"这样的简单需求，生成包含景点、酒店、餐饮的基础行程方案。但很快我们发现了几个明显的问题：首先，用户体验很不智能，每次都要重复说自己的偏好，基础信息，系统没有记忆能力；其次，当用户问一些平台政策类问题，比如"平台的优惠政策"，纯粹靠大模型容易产生幻觉；第三，意图识别比较简单，采用关键词匹配的方式，准确率只有65%，复杂意图识别不出来。

针对这些问题，我们进行了第一轮优化。我设计了一套两层记忆系统来解决用户偏好记忆的问题。短期记忆基于滑动窗口机制，保存当前会话的最近10轮对话用于理解上下文；长期记忆则采用JSON文件进行持久化存储（生产环境规划迁移到PostgreSQL），保存用户偏好（酒店品牌、航空公司、座位偏好等）、历史行程记录、完整聊天历史，实现跨会话的数据共享。同时我还加入了一个PreferenceAgent，基于大模型的语义理解能力，它能够智能识别用户是在追加偏好（比如"我还喜欢如家"）还是覆盖偏好（比如"我搬家到上海了"），让系统真正能记住用户的习惯。这个改进上线后，用户偏好记忆准确率达到了95%左右，基本不需要重复输入了。同时我还引入了异步LLM总结机制，定期对历史对话和行程进行自动摘要，进一步提升了记忆的可用性。

后来为了解决知识支撑不足的问题，我们又做了第二轮优化，实现了一个RAG知识库系统。我采用了本地化部署的方案：用Milvus Lite作为向量数据库（单机部署，数据存储在milvus_lite.db文件），BGE-small-zh-v1.5做Embedding模型（本地加载，离线可用），把公司内的差旅相关文档（优惠政策、城市攻略、常见问题FAQ等8类文档）都做了向量化存储。在文档处理环节实现了Chunking策略，针对长文档采用段落级别的滑动窗口切分（每段最多600字符，overlap 100字符），保证检索精度。当用户咨询政策类问题时，系统会先通过余弦相似度检索相关文档片段（top_k=3），再让大模型基于检索内容生成回答，同时返回文档来源实现可追溯性，大大增加了可信度。另外，我们还发现用户经常需要查询实时信息，比如"杭州明天天气怎么样"这类问题。针对这个需求，我设计了InformationQueryAgent的Skill，集成wttr.in天气API和DDGS网络搜索引擎，能够联网获取实时信息并用大模型自动摘要。这样系统既能基于知识库回答规范性问题（如差旅政策、报销标准等静态知识），又能处理实时性查询（如天气、景点实况等动态信息），功能更完善了。

接下来我们重构了意图识别和Agent调度模块，这是第三轮优化的重点。最初的IntentionAgent只是简单的关键词匹配，我做了一轮测试集评估，收集了1000个真实用户输入，发现准确率只有65%。主要问题在于无法处理同义表达、复合意图和上下文依赖。于是我们把意图识别升级成基于LLM的语义理解系统（底层调用豆包Seed模型），能够识别6大类意图：规划行程（itinerary_planning）、查询记忆（memory_query）、管理偏好（preference）、知识问答（rag_knowledge）、信息查询（information_query）、事项收集。同时我还实现了一套动态的Agent调度机制，采用"优先级+并行"的混合模式。具体来说，根据意图识别结果动态分配优先级，同优先级的Agent通过asyncio.gather并行执行，不同优先级按依赖关系串行执行。比如用户说"我想3月11日从北京去杭州出差一周"，系统会并行调用EventCollectionAgent（提取结构化信息）和MemoryQueryAgent（查询历史偏好），然后再调用ItineraryPlanningAgent整合生成行程。这个并行调度机制让整个流程从串行的30秒优化到15秒，响应速度快了一倍，意图识别准确率也提升到了90%以上。

在架构设计层面，我还做了一些工程化的改进。首先是实现了插件化的Skill Plugins架构，参考了Claude Code的设计思路，把所有子Agent重构成独立的Skill插件。每个Skill是一个独立目录，包含SKILL.md配置文件和script/agent.py实现代码。系统启动时LazyAgentRegistry自动扫描.claude/skills目录，发现并注册所有可用的Agent插件，但只做发现不做加载。只有当某个Agent真正被调用时，才通过importlib.util动态导入模块并实例化，加载后缓存起来，下次就不用重复加载了。这样做的好处是新增功能非常简单，每个Skill可以独立开发、测试和部署，未使用的Skill不会被加载进内存，项目启动速度优化到了3秒以内。在生产环境规划中，我还会引入Redis作为缓存层来优化热数据访问性能，将短期记忆（会话状态）、用户偏好热数据、LLM总结结果都缓存到Redis中（短期记忆TTL设置为1小时），与PostgreSQL形成"缓存+持久化"的互补架构。


目前这个智能旅行助手系统已经能够支持完整的行程规划流程，覆盖意图识别、事项收集、偏好管理、知识问答、信息查询、行程生成这六大功能模块。整体来看，这套系统让普通用户可以像有专业旅行顾问一样，快速获取个性化的行程建议和全方位的旅行信息，大大降低了旅行规划的门槛。




项目问题
1. 简单说说AgentScope是什么框架
AgentScope是阿里开源的多智能体开发框架。简单说就是一个让多个Agent协作干活的框架，核心是Actor模型的消息传递——每个Agent是独立的Actor，通过Msg对象进行通信，天然支持异步和并行。它内置了ReAct、记忆管理、人机协同这些能力，5分钟就能快速上手。在这个项目里我主要看中了它的消息传递架构和异步支持，后面详细说。

2. 为什么选择用AgentScope，不用LangGraph
AgentScope的特点：
- 基于消息传递的架构：采用Actor模型的分布式框架，Agent之间通过Msg对象进行清晰的消息交互
- 开箱即用更丰富：内置ReAct、记忆管理、人机协同、语音交互等功能，5分钟快速上手
- 多Agent协作更友好：专注于多Agent系统，提供灵活的消息中心和调度机制
- 生产就绪：实时控制、中断处理、沙箱执行等企业级能力完善，内置AgentStudio可视化工具
在这个项目里
1. 消息传递架构适合多Agent协作。 AgentScope采用Actor模型，Agent之间通过Msg对象传递结构化数据。我们系统有6个子Agent，需要频繁传递意图识别结果、调度决策、上下文信息，消息传递的方式让每个Agent的输入输出都很清晰，调试时能直观看到整个数据流向。LangGraph是基于全局State的状态图架构，Agent数量多的时候状态管理容易变得臃肿。
2. AgentBase基类适合插件化架构。 所有子Agent继承AgentBase，统一暴露reply方法。这样我们的懒加载注册器可以动态扫描目录、自动发现并实例化Agent插件，新增功能只需写一个继承AgentBase的类，无需修改核心代码。LangGraph的节点需要手动定义状态Schema和边的转换逻辑，做不到这种即插即用。
3. 异步支持适合并行调度。 我们实现了"优先级+并行"的混合调度——同优先级Agent并行执行，不同优先级串行执行。AgentScope的异步reply接口配合asyncio实现并行非常自然，把响应时间从30秒降到了15秒。LangGraph的并行需要在图结构中预定义，运行时动态调整并行策略比较困难，而我们哪些Agent需要并行完全取决于意图识别的实时结果。

补充一句客观的：LangGraph当然也很优秀，它的状态图可视化调试很直观，适合Agent数量少、流程固定的场景。但我们的需求是6个Agent动态组合、运行时决定并行策略，所以AgentScope更合适。

3. 简单介绍一下你设计的几个Skills
项目中共实现了6个Skills，按功能可以分三层：

信息提取层：
1. event-collection(事项收集):负责从用户输入中提取结构化信息，如出发地、目的地、日期、时长等关键要素。它能够处理相对时间表达(如"明天""2月27日")，自动补全缺失信息，特殊场景识别(如"一日游"自动设置duration_days=1)，为行程规划提供基础数据。
2. preference(偏好管理):管理用户的旅行偏好，包括酒店品牌、航空公司、座位偏好等。它能够智能识别用户是在追加偏好(比如"我还喜欢如家")还是覆盖偏好(比如"我搬家到上海了")，通过动态读取当前已保存的偏好并与新输入对比分析，实现精准的偏好更新。
3. memory-query(记忆查询):基于用户的长期记忆回答历史相关问题。它能够查询用户旅行历史(trip_history)、用户偏好(preferences)和历史对话记录(chat_history)，使用LLM基于记忆生成自然语言回答。

知识检索层：
4. ask-question(知识问答):基于Milvus Lite向量数据库和BGE-small-zh中文向量模型，实现差旅政策、城市攻略等企业知识的智能检索和问答。采用向量检索，支持top_k=3的文档召回，结合相似度过滤和文档溯源机制防止幻觉。
5. query-info(信息查询):集成wttr.in天气API和DDGS网络搜索引擎，让系统具备联网查询能力。对于天气类问题优先使用wttr.in获取可靠结果，其他问题通过DDGS搜索并开启safesearch过滤可疑来源。

决策生成层：
6. plan-trip(行程规划):整合前面所有Skill的结果，生成完整的个性化行程方案。它从previous_results中提取其他Skill的数据，结合用户偏好信息和当前时间季节信息，通过动态读取Skill配置中的详细指令(Progressive Disclosure)，调用LLM生成包含景点推荐、酒店预订、交通安排等的完整行程计划。

面试时建议先说三层架构的概括，面试官追问哪个再展开。

4. 这个项目的核心指标有哪些，每个指标是怎么算出来的
这个项目在性能、准确性和用户体验三个维度都有明确的量化指标，以下是核心指标及其计算方法（每行一个指标）：
- 意图识别准确率90%+： 在1000条测试集上，统计IntentionAgent识别正确的意图数量除以总样本数，从最初基于关键词匹配的65%提升到基于LLM语义理解的90%+
- 知识库问答准确率95%： 在RAG知识库的测试集中，统计RAGKnowledgeAgent返回的答案与标准答案一致的比例，通过向量检索（余弦相似度）和相似度阈值过滤实现
- 用户偏好记忆准确率95%： 统计PreferenceAgent正确识别追加/覆盖动作的成功率，在测试集中对比系统保存的偏好与用户实际意图的一致性
- 系统响应时间从30秒优化到15秒： 从用户发送query到返回完整结果的端到端耗时，通过引入优先级并行调度机制，让Priority 1的多个信息收集Agent并行执行而非串行，响应时间减半
4.1 介绍下1000条测试集
- 测试集来源：从公司内部旅行系统日志中抽取真实用户查询，经过脱敏处理后筛选出1000条。
- 测试集内容：覆盖5大类意图(行程规划、偏好管理、记忆查询、知识问答、信息查询)，每类约200条。包含口语化表达("明天去杭州")、同义表达("订酒店"vs"订宾馆")、上下文依赖("那边天气怎么样")、复合意图("我要去北京，喜欢住汉庭")等情况。
- 测试集格式（以意图识别为例，其他测试ground_truth替换为对应的真实标签）：
{
  "query": "我要3月11日从北京去上海,喜欢住汉庭"，
  "ground_truth": ["itinerary_planning"，"preference"]
}
- 如何判断是否正确：都是人工标注的
  - 意图识别准确率：对1000条测试集，逐条调用IntentionAgent，提取输出JSON中的意图类型列表，与标注对比。完全匹配算对，否则算错。准确率 = 正确数 / 1000
  - 偏好记忆准确率：从1000条中筛选出约200条偏好相关的query(标注包含"preference"意图的)，让系统实际执行完整流程，检查最终写入长期记忆的偏好数据是否正确(比如"我还喜欢如家"应该追加到hotel_brand列表，"我搬家到上海了"应该覆盖home_location字段)
4.2 整个项目的token消耗（示例）
一次完整行程规划查询：≈ 4,500–6,000 Token。其中：输入 ≈ 4,000–5,000 Token，输出 ≈ 500–1,000 Token
具体拆分为：
1. 系统提示 + Skill 元数据
  - 约 800–1,200 Token
  - 用了渐进式披露，只加载元数据，不加载全量指令
2. 意图识别 Agent（IntentionAgent）
  - 输入：用户 query + 上下文 + 技能清单
  - 输出：结构化意图 + 调度计划
  - 单次：≈ 600–900 Token
3. 并行子 Agent 调用（Priority 1 并行）
  - 事项收集、偏好查询、RAG、信息查询 并行执行
  - 每个子 Agent：≈ 300–500 Token / 个
  - 一次通常跑 3–5 个，合计：≈ 1,200–2,000 Token
  - 因为并行 + 结果摘要，不把全量中间结果塞上下文，节省大量 Token
4. 记忆系统（短期 + 长期总结）
  - 短期记忆：最近 10 轮对话 ≈ 300–500 Token
  - 长期记忆总结：异步生成，缓存复用 ≈ 200 Token
  - 合计：≈ 500–700 Token
5. 最终行程生成输出
  - 行程方案 + 说明
  - 输出：≈ 500–1,000 Token
按当时我们用的豆包模型价格算，一次完整查询大概花费几毛钱到一块钱左右。这个成本对于内部工具来说是可以接受的，但如果面向C端用户，还需要进一步压缩，比如通过缓存高频问题的结果来减少LLM调用。
4.3 哪些操作优化了token使用
- 并行调度，避免多 Agent 上下文累积
- 子 Agent 结果先摘要再聚合，不塞全量返回
- 渐进式披露，只加载 Skill 元数据
- 记忆异步总结 + 缓存，避免重复总结耗 Token
- RAG 只召回 top3，控制注入长度

5. 这个项目采用的是什么Agent架构，为什么选择这个架构
这个项目采用的是Plan-and-Execute架构，这是一种先规划再执行的两阶段架构模式。
Plan-and-Execute架构的核心思想是将Agent的工作分成两个阶段：Planning阶段由IntentionAgent负责理解用户意图并生成完整的执行计划，Execute阶段由调度器（OrchestrationAgent）按照计划调度各个子Agent执行具体任务。这种架构的优势在于规划和执行解耦，IntentionAgent专注于理解和规划，调度器专注于调度和执行，职责清晰。
选择Plan-and-Execute而非ReAct的原因主要有三点：
第一是任务复杂度，我这个旅行助手项目需要协调多个子Agent（事项收集、偏好管理、知识问答、信息查询、行程规划等），如果用ReAct的Thought-Action-Observation循环，每次只能执行一个Action然后观察结果再决定下一步，效率太低。而Plan-and-Execute可以一次性规划好所有需要调用的Agent及其优先级，然后并行执行，大大提升了效率。
第二是成本控制，ReAct架构需要多轮调用LLM进行推理，每轮都要消耗token，而Plan-and-Execute只需要IntentionAgent调用一次LLM生成计划，后续执行不需要反复调用LLM做决策，token消耗更少。
第三是可预测性，ReAct的循环次数不确定，可能陷入死循环或者过早终止，而Plan-and-Execute的执行流程是确定的，IntentionAgent生成的计划明确指定了要调用哪些Agent、调用顺序和优先级，执行过程可控可预测。

6. 意图识别Agent的核心逻辑是什么
IntentionAgent是整个系统的核心，继承自AgentScope的AgentBase基类，负责理解用户输入并生成Agent调度计划。核心逻辑包括四个步骤：
第1步：推理过程生成 - 分析用户query的核心诉求，识别关键实体和意图信号，判断是否需要结合对话历史进行消歧。
第2步：多意图识别 - 识别所有可能的用户意图（可以是多个），为每个意图分配置信度（0-1之间），说明识别该意图的原因。
第3步：智能Query改写 - 识别口语化表达并进行标准化，补全省略的上下文信息，提取和重组关键信息。当用户说"明天去杭州"时，Prompt中注入了当前时间（年月日时分+星期），大模型会自动把"明天"推算成具体日期。
第4步：构建结构化决策 - 基于识别的意图，决定调用哪些子Agent，说明调用顺序和优先级。最后调用LLM（底层用豆包Seed模型）进行意图识别，要求输出严格的JSON格式，包含reasoning（推理过程）、intents（意图列表）、key_entities（关键实体）、rewritten_query（改写后的查询）和agent_schedule（Agent调度计划）。
在实现上，我构建了一个详细的Prompt，包含当前时间、对话历史、可调度的子Agent信息等上下文。
这里我采用了插件化的Skill架构，所有的子Agent都是作为独立的Skill插件实现的（如ask-question、memory-query、plan-trip等），存放在.claude/skills目录下。
因此，IntentionAgent需要动态获知当前系统中有哪些可用的Skill。我通过SkillLoader动态读取每个Skill的SKILL.md文件，提取YAML frontmatter中的元数据（名称、描述、触发条件），然后将这些信息注入到Prompt中，让LLM能够了解所有可用的Agent选项。
这样设计的好处是：新增或删除Skill时，IntentionAgent会自动感知到变化，无需手动修改Prompt。
同时我还实现了Progressive Disclosure（渐进式暴露）机制，在意图识别阶段只通过get_skill_prompt()加载元数据列表，不加载详细的执行指令；在具体Agent执行时再通过get_skill_content()加载完整的SKILL.md内容，避免Prompt过长。
JSON解析方面，考虑到LLM输出可能不标准，我实现了6层降级解析策略：直接parse -> 去除控制字符 -> 修复单引号 -> 修复尾随逗号 -> 智能处理换行转义 -> json5库兜底。如果所有策略都失败，fallback到默认调度plan。

7. 你刚才提到Progressive Disclosure，详细说下它是什么
Progressive Disclosure（渐进式披露）是一种信息架构设计模式，核心思想是"按需加载，分层披露信息"，避免一次性加载所有信息导致的复杂度和token开销。
在这个项目中，我参考了Claude Code的Agent Skills设计，每个Skill是一个独立目录，包含SKILL.md配置文件和script/agent.py实现代码。我实现了三层信息结构：
- 第1层：元数据层（Metadata） - SKILL.md的YAML frontmatter，包含Skill名称和简短描述。在意图识别阶段通过SkillLoader.get_skill_prompt()加载，生成编号列表注入到Prompt。这让IntentionAgent能够了解所有可用的Agent，但不会被详细指令淹没。
- 第2层：指令层（Instructions） - SKILL.md中frontmatter之后的详细指令内容。在具体Agent执行时通过SkillLoader.get_skill_content()按需加载。例如RAGKnowledgeAgent执行时，会动态读取RAG生成回复的要求，注入到Prompt中。
- 第3层：资源层（Resources） - script/agent.py代码实现、数据文件等，在实际使用时才通过importlib动态导入。
这种设计的好处是：首先节省token，意图识别阶段只需要加载元数据，6个Skill的元数据加起来也就几百个token；其次提升准确性，每个Agent只看到自己需要的指令，不会被无关信息干扰；最后易于维护，每个Skill的说明和指令都在自己的SKILL.md文件中，修改时不影响其他部分。

8. 你提到了优先级和并行调度，详细说一下
这是系统性能优化的关键设计。在最初版本中，所有Agent都是串行执行的，比如意图识别→事项收集→偏好管理→行程规划。
后来我重构了调度机制，在OrchestrationAgent中引入了"优先级+并行"的混合模式。
优先级分组：
- Priority 1（并行执行） - 信息收集类Agent，包括memory_query、event_collection、preference、information_query、rag_knowledge。这些Agent都是信息获取，互不依赖，可以并行执行。
- Priority 2（依赖Priority 1） - 行程规划类Agent（itinerary_planning），需要使用Priority 1收集的信息。
实现细节：OrchestrationAgent的reply方法中，首先从IntentionAgent的输出中提取agent_schedule，按priority字段排序。然后按优先级分组，当优先级变化时，执行上一批次的所有task。单个task直接执行，多个task使用asyncio.gather(*coroutines, return_exceptions=True)并行执行。这样Priority 1的多个Agent可以同时并行调用，大大缩短了总体执行时间。需要注意的是，多个Agent全并行时，实际耗时取决于最慢的那个，所以我们设置了超时机制兜底。
实际测试中，响应时间从串行的30秒优化到了15秒，提升了一倍。

9. 调度器的核心职责是什么
调度器（OrchestrationAgent）是整个多Agent系统的调度中枢，负责协调各个子Agent的执行。它的核心职责包括：
1. 解析意图识别结果 - 从IntentionAgent的输出中提取agent_schedule（调度计划），获取每个Agent的调用信息、优先级、调用原因等。
2. 准备上下文信息 - 构建统一的上下文，包括意图识别的reasoning、intents、key_entities、rewritten_query，从MemoryManager获取的短期记忆（最近10轮对话），从LongTermMemory获取的用户偏好。
3. 按优先级调度Agent - 按priority排序task，同优先级的并行执行，不同优先级顺序执行。执行单个Agent时，会构造统一格式的输入Msg。
4. 聚合执行结果 - 将所有Agent的执行结果聚合成统一的JSON结构，包含status、intention、agents_execed、results（每个agent的name、priority、status、data）等字段。
5. 更新记忆系统 - 将重要信息写入长期记忆：如果是PreferenceAgent，保存用户偏好到LongTermMemory（支持append和replace两种模式）；如果是ItineraryPlanningAgent，保存行程历史到LongTermMemory（从event_collection结果中提取出发地、目的地等结构化信息）。
整个流程清晰有序，每个子Agent只需关注自己的业务逻辑，不需要知道调度细节。

10. 多Agent之间如何完成通信
这个项目采用了AgentScope框架的消息传递机制，所有Agent之间通过标准化的Msg对象进行通信。
- Msg对象的结构包含三个核心字段：name表示发送者的Agent名称，content是消息内容通常是JSON格式的结构化数据，role标识角色类型如user或assistant。这种统一的消息格式让所有Agent都能理解彼此的输出。
- 调度器作为消息中枢，负责在各个子Agent之间传递消息。具体流程是：首先调度器从IntentionAgent接收意图识别结果，然后为每个子Agent构造标准的输入context字典，这个context包含上下文信息、调用原因、期望输出、以及前序Agent的执行结果。子Agent处理完后返回一个新的Msg对象，调度器解析这个返回的Msg，提取结果并聚合到一起。
- 并行通信的实现：采用异步机制，对于同优先级的多个Agent，调度器会同时创建协程任务，通过asyncio.gather并发执行，每个Agent独立处理自己的输入并返回结果，调度器统一等待所有协程完成后收集结果。return_exceptions=True确保即使某个Agent抛出异常，其他Agent也不会被影响。
- 上下文共享机制确保所有Agent都能访问必要的信息。调度器在准备上下文时，会从记忆系统获取短期记忆和用户偏好，然后把这些信息打包到context中传递给子Agent。如果某个Agent需要知道前面Agent的执行结果，调度器会把之前所有Agent的输出也包含在context的previous_results字段中，实现信息的流转和共享。
这种基于消息传递的通信方式，相比传统的函数调用或共享状态，具有更好的可扩展性和容错性，符合分布式系统的设计原则。

11. 你刚才提到的记忆系统，详细说一下设计思路
记忆系统是实现用户偏好记忆的核心，我设计了一套两层记忆架构：短期记忆+长期记忆。
短期记忆（ShortTermMemory）：
- 存储：纯内存的双端队列（collections.deque），保存最近10轮对话（max_turns=10，即max_messages=20条消息）
- 生命周期：会话级，会话结束后清空
- 每条消息包含：role、content、timestamp（ISO格式）、metadata
- 滑动窗口：当消息超过max_messages时，自动丢弃最早的：self.messages = self.messages[-max_messages:]
- 用途：为当前对话提供上下文，让Agent理解用户的连续提问
长期记忆（LongTermMemory）：
- 存储：JSON文件持久化（data/memory/{user_id}.json），每个用户一个文件
- 生命周期：跨会话，永久保存
- 数据结构：包含user_id、preferences（偏好列表）、chat_history（全量聊天记录）、trip_history（历史行程）、statistics（统计信息）
- 用途：记住用户的长期偏好和历史行为，实现个性化推荐
- 数据迁移：内置_migrate_data方法处理格式迁移，从旧版字典格式的preferences迁移到新版列表格式，保证数据兼容
异步LLM总结：
- 为了避免长期记忆过多导致token爆炸，我引入了异步LLM总结机制
- get_long_term_summary_async方法对跨会话的聊天历史和行程历史进行总结，生成200字以内的摘要
- 总结内容包括：用户的旅行偏好、曾经问过的重要问题、旅行历史、其他相关上下文
- 摘要作为system消息传给IntentionAgent，提供跨会话的上下文
两层记忆配合：MemoryManager统一接口，add_message同时写入短期和长期记忆，get_context_for_agent组装短期记忆（最近5轮对话）+长期记忆总结，一次性提供给Agent使用。

12. 为什么选择PostgreSQL而不是MySQL
当前版本使用JSON文件做持久化存储，因为项目还处于原型验证阶段，JSON文件足够支撑开发测试。生产环境规划中选择了PostgreSQL，主要基于以下考虑：
PostgreSQL的核心特点：
1. 对象-关系型数据库：PostgreSQL不仅是关系型数据库，还支持面向对象特性，扩展性更强
2. 严格遵循SQL标准：支持复杂查询、窗口函数、CTE等高级特性
3. MVCC并发控制：多版本并发控制，读写互不阻塞，并发性能优异
4. 丰富的数据类型：原生支持JSON/JSONB、数组、UUID等，适合存储半结构化数据
相比MySQL的优势：
1. 复杂查询性能更好：查询优化器更先进，多表JOIN、子查询等场景性能优于MySQL
2. JSON支持更强大：JSONB类型支持索引和高效查询，可以像NoSQL一样操作JSON数据，适合存储用户偏好这种半结构化数据
3. 并发写入能力更强：MVCC机制下高并发写入不会出现锁竞争
4. 数据一致性保证更好：故障恢复、数据完整性约束更严格

13. 你提到还引入Redis缓存，详细说一下
当前版本短期记忆使用内存存储，生产环境规划中设计了"Redis缓存+PostgreSQL持久化"的互补架构：
Redis缓存的三类数据：
1. 短期记忆（会话状态） - TTL设置为1小时，支持分布式部署时的会话共享
2. 用户偏好热数据 - 缓存用户的常用偏好（酒店品牌、航空公司等），减少PostgreSQL的高频查询
3. LLM总结结果 - 对长期记忆的LLM总结结果进行缓存，避免重复调用LLM。Key格式：user:{user_id}:summary，TTL设置为30分钟
缓存更新策略：
- Write-Behind：写入时先更新Redis，异步刷入PostgreSQL——偏好更新不是强一致场景，异步落库即可
- 数据Lazy Loading：读取时先查Redis，未命中再查PostgreSQL并写入Redis
- TTL过期：短期记忆1小时过期，偏好和总结根据更新频率设置不同TTL
性能提升预期：并发能力从100 QPS提升到500+ QPS

14. 插件化的Skill Plugins架构是怎么实现的
我参考了Claude Code的设计思路，实现了一套完整的插件化架构，让新增Agent变得非常简单。
LazyAgentRegistry（懒加载注册器）：
工作机制分2步：
1. 自动发现 - 系统启动时扫描.claude/skills目录，找到所有包含script/agent.py的Skill目录，记录下"这个Skill在哪"，但这时候不加载代码。通过_legacy_mapping维护目录名和内部Agent名称的映射（如rag_knowledge -> ask-question）
2. 懒加载（按需加载） - 只有当某个Agent真正被调用时，才去加载。通过__getitem__触发，使用importlib.util.spec_from_file_location动态导入模块，插入project_root到sys.path确保跨模块导入可用，然后用inspect.getmembers找到AgentBase的子类并实例化。如果Agent的__init__签名中有memory_manager参数，自动注入MemoryManager实例。加载后缓存到self.cache，下次就不用重复加载了
这种设计的好处：
- 零配置：不需要手动注册Agent，系统自动发现
- 热插拔：新增或删除Skill不影响核心代码，只要在.claude/skills下新建目录放SKILL.md和script/agent.py就行
- 独立开发：每个Skill都是自包含的，可以独立开发和测试
- 快速启动：懒加载机制让系统启动时只扫描不加载，启动速度优化到3秒以内

15. RAG知识库是怎么实现的
RAG（检索增强生成）知识库是RAGKnowledgeAgent的核心功能。
知识库中存储的文档包括8类：差旅标准与规定（01_travel_standards.txt）、报销政策（02_reimbursement_policy.txt）、预订指南（03_booking_guide.txt）、常见问题FAQ（04_faq.txt）、应急程序（05_emergency_procedures.txt）、平台使用指南（06_platform_guide.txt）、城市特定攻略（07_city_specific_tips.txt）、环保计划（08_environmental_initiatives.txt）。
采用BGE-small-zh-v1.5作为Embedding模型（本地加载，512维向量，离线可用），采用Milvus Lite向量数据库（单机文件存储：milvus_lite.db），集合名称business_travel_knowledge，使用余弦相似度（COSINE）作为距离度量。
针对长文档采用段落级别的滑动窗口切分（split_text函数，每段最多600字符，overlap 100字符），先按段落空行分割，合并段落直到达到max_chars限制，如果单个段落超过限制就硬分割。为每个chunk添加metadata（类别、来源文档等），保证检索精度和上下文完整性。
检索时采用向量检索，top_k=3（可配置），通过milvus_client.search()检索最相关的文档片段。将检索到的内容填入到prompt里交给大模型生成回复。如果未找到相关知识，直接返回"抱歉，我在知识库中没有找到相关信息"，不让大模型瞎猜。
RAG Agent还内置了连接保障机制：_ensure_connection方法在连接失败时自动重连，提升系统鲁棒性。
知识库问答准确率达到95%，基本能够准确回答差旅政策、报销标准等企业知识问题。

16. 向量数据库为什么用Milvus
项目中实际使用的是Milvus Lite（单机版，pip install就能用），选择它的原因：
1. 轻量级部署：Milvus Lite是单机版，不需要部署服务端，pip install pymilvus[milvus-lite]就能用，数据存在一个本地.db文件里，非常适合原型验证和中小规模场景
2. 原生向量数据库：相比FAISS等纯算法库，Milvus Lite自带持久化、元数据过滤、标量+向量混合查询能力，不用自己拼装这些基础设施
3. 向量+标量混合检索：支持过滤条件+相似度联合查询，可先通过标量过滤缩小检索范围，再做向量相似性检索，避免全量向量检索的性能损耗
4. 平滑升级路径：如果后续数据量和并发量上去了，可以从Milvus Lite平滑迁移到Milvus服务器版（分布式架构，支持水平扩展、分片、副本），不需要重写业务代码
面试时注意：当前版本用的是Milvus Lite单机部署，如果说成分布式集群就不准确了。

17. 召回用的是什么检索算法
项目中当前版本采用的是向量检索（语义匹配）：
使用余弦相似度（COSINE）作为相似度算法。用户问题通过BGE-small-zh-v1.5 Embedding模型转化为512维向量后，计算与知识库中文档片段向量的余弦相似度。余弦相似度只关注向量的方向，不考虑长度，非常适合文本语义匹配。优势是能够理解语义，即使用户换一种说法，只要意思相近就能召回。
检索流程：
1. 用户问题通过Embedding模型转化为向量
2. 通过milvus_client.search()进行向量检索，召回top_k=3个最相关的文档片段
3. 将检索到的文档片段作为上下文注入到Prompt中，让大模型生成回答
生产环境规划中会引入BM25关键词匹配做混合检索（Hybrid Search），结合向量检索和关键词匹配的优势，采用RRF（Reciprocal Rank Fusion）做结果融合，进一步提升召回准确率。

18. 意图识别准确率从65%提升到90%+，是怎么做到的
主要通过三个方面的改进：
1. 从关键词匹配升级到语义理解：最初的IntentionAgent只是简单的关键词匹配，例如看到"去"就认为是行程规划，看到"喜欢"就认为是偏好管理。但这种方式无法处理同义表达和复杂意图。我重构后采用基于LLM的语义理解（底层调用豆包Seed模型），构建了详细的Prompt，包含6大类意图的详细定义和触发条件、意图消歧规则（如"我去过北京吗？"→memory_query而非information_query）、当前时间信息（用于推算相对日期）、以及Skill元数据列表（让LLM知道有哪些可用Agent）。
2. 引入多意图识别：支持一个用户输入同时包含多个意图，例如"我想从北京去上海，喜欢住汉庭"同时包含行程规划和偏好管理。每个意图都有confidence置信度，IntentionAgent会根据置信度决定是否调用对应的Agent。
3. 利用对话历史消歧：引入短期记忆和长期记忆总结，让IntentionAgent能够理解上下文。短期记忆保留最近10轮对话（截断每条消息到800字符防止过长），长期记忆总结提供跨会话的用户偏好和历史行为摘要。
JSON解析鲁棒性：LLM输出的JSON格式可能不标准（缺少引号、尾随逗号、控制字符等），我实现了6层降级解析策略（直接parse→去除控制字符→修复单引号→修复尾随逗号→智能换行处理→json5库兜底），确保即使输出不规范也能正确提取意图。如果所有策略都失败，fallback到默认的information_query意图。

19. 你的对话Agent如何处理超出上下文窗口的长对话
在当前实现中主要通过两层记忆+LLM总结来处理：
1. 滑动窗口机制：
  - ShortTermMemory使用内存列表实现固定长度的滑动窗口
  - max_turns=10，只保留最近10轮对话（即max_messages=20条消息）
  - 当对话超过max_messages时，自动丢弃最早的：self.messages = self.messages[-max_messages:]
2. LLM异步总结：
  - 在每次对话前，调用get_long_term_summary_async方法
  - 对长期聊天历史（排除当前会话，取最近50条消息）和行程历史进行总结
  - 使用LLM生成200字以内的摘要，作为system消息传给IntentionAgent
  - 总结内容包括：旅行偏好、重要问题、旅行历史、其他相关上下文
3. 分层记忆管理：
  - 短期记忆提供当前会话的连贯性（约300-500 tokens）
  - 长期记忆总结提供跨会话的上下文（压缩到200 tokens以内）
  - IntentionAgent接收的消息列表包含：system（长期记忆总结）+ user/assistant（最近5轮对话的短期记忆上下文）
这样既保证了上下文连贯性，又控制了token消耗在合理范围内。

20. 你做这个项目遇到过什么困难/挑战
主要遇到了三个方面的困难：
1. Agent调度的复杂性：最初实现串行调度时，逻辑比较简单。但引入优先级和并行调度后，需要处理很多边界情况：
  - 如何保证同优先级Agent真正并行执行——最终用asyncio.gather配合return_exceptions=True实现，既保证了并行又防止单个Agent异常影响其他Agent
  - 如何区分界定意图之间的区别——比如"北京南站的酒店多少一晚"是information_query（问客观信息）还是memory_query（问自己的历史），需要在IntentionAgent的Prompt中加入消歧规则，反复调试Prompt才达到90%+的准确率
  - 并行执行时最慢的Agent决定整体耗时，后来引入了超时机制兜底
2. 记忆系统的设计权衡：两层记忆系统在设计时面临很多选择，有一个具体的坑：短期记忆保留多少轮这件事，我一开始设的是5轮，结果发现用户连续追问行程细节的时候，第6轮就开始丢失上下文了，Agent不知道用户说的"那个酒店"指的是哪个。后来改成15轮，又发现token消耗太大，IntentionAgent单次调用多了将近200 token。最后折中到10轮，配合异步LLM总结把更早的历史压缩成200字摘要，既保证了上下文连贯，又控制了token成本。
  - 用户偏好的追加和覆盖逻辑（PreferenceAgent的智能识别）也是个难点——"我还喜欢如家"是追加，"我搬家到上海了"是覆盖，需要LLM准确理解语义
3. 插件化架构的实现：
  - 动态模块加载——用importlib.util动态导入，需要处理sys.path注入确保跨模块import可用
  - 依赖注入——通过inspect检查Agent的__init__签名，自动注入memory_manager
  - 名称映射——Skill目录名（如ask-question）和内部Agent名称（如rag_knowledge）不一致，通过_legacy_mapping做桥接
  最终实现了一个清晰优雅的插件系统，新增Agent只需在.claude/skills下新建目录放SKILL.md和script/agent.py就行。

21. 如果你的Agent执行失败了，系统会怎么处理
我设计了多层容错机制来保证系统的鲁棒性。
首先在Agent级别，每个Agent的执行都有异常捕获。在OrchestrationAgent中，通过asyncio.gather(*coroutines, return_exceptions=True)实现并行执行，即使某个Agent抛出异常，其他Agent也会继续运行，不会互相影响。执行失败的Agent会返回error状态，但不中断整个流程。
其次是重试机制。在utils/llm_resilience.py中实现了retry_with_backoff函数，对于可重试的异常（超时、429速率限制、5xx服务器错误、ConnectionError、OSError），采用指数退避策略自动重试：delay = min(base_delay * 2^attempt, max_delay)，最多重试3次。配置上base_delay=1秒，max_delay=30秒。
第三是熔断器机制（Circuit Breaker）。在utils/circuit_breaker.py中实现了熔断器模式：初始状态CLOSED（正常请求），连续失败5次后进入OPEN状态（拒绝所有请求），60秒后进入HALF_OPEN状态（允许试探性请求），连续成功2次后恢复CLOSED。这样在下游服务（如LLM API）不可用时，系统能快速失败而不是持续等待超时。
第四是部分失败处理。如果多个Agent并行执行时有的成功有的失败，系统不会全部丢弃，而是会标记为部分成功，把成功的结果返回给用户，让用户至少能看到一部分信息。
系统不会因为单个Agent的问题就完全崩溃，而是优雅降级，尽可能提供部分功能。

22. 你提到了重试机制，为什么不是所有异常都重试
重试机制需要谨慎设计，不是所有异常都应该重试。
我的判断标准是看这个错误重试后有没有可能成功。在llm_resilience.py中，我只对以下异常进行重试：超时、429速率限制、5xx服务器错误、ConnectionError、OSError——这些都是临时性的问题，等一会儿重试就可能成功。但如果是认证失败、请求格式错误、业务逻辑错误这种，重试多少次都不会成功，就不应该浪费资源去重试。
另外重试也不是立即重试，而是采用指数退避策略：delay = min(base_delay * 2^attempt, max_delay)。第一次等1秒，第二次等2秒，第三次等4秒，最多等30秒。这样做有两个好处：一是给下游服务恢复的时间，二是避免大量请求同时重试反而加剧服务压力。
最多重试3次，如果还失败就直接返回错误。这样既保证了系统可靠性，又避免了无效重试带来的资源浪费。

23. 你的RAG系统如何保证不会产生幻觉
大模型幻觉是AI应用的核心问题，我通过四层防护机制来控制。
第一层是Prompt强约束，在RAGKnowledgeAgent的SKILL.md中明确规定必须严格基于知识库内容回答，不得编造信息，如果知识库中没有相关信息就明确告诉用户"找不到"。
第二层是RAG知识增强，先通过向量检索（余弦相似度，top_k=3）找到相关的文档片段，然后把这些片段明确标记为知识库信息传给大模型，要求它仅基于这些内容生成答案。
第三层是相似度过滤，只使用top3最相关的文档，如果检索结果的相关度不够，直接返回"我在知识库中没有找到相关信息"，而不是让大模型瞎猜。
第四层是文档溯源，返回结果时会标注信息来源文档，用户可以查看和验证答案的准确性，如果发现问题可以追溯到具体文档进行优化。
通过这四层机制，RAG知识库问答的准确率达到95%，把幻觉的程度控制在了可接受的范围内。

24. 你的系统如何处理用户的模糊时间表达
这是IntentionAgent的一个重要功能。用户经常说"明天去杭州"、"下周二出发"、"2月28日开始"这种模糊的时间表达，如果不处理，后续的Agent就无法准确理解具体日期。
我的解决方案是在IntentionAgent的Prompt中注入当前时间信息，包括年月日时分和星期几（datetime.now()获取），然后明确告诉大模型"当用户说明天或者相对时间时，请根据当前时间推断出完整的日期"。这样IntentionAgent在分析用户意图时，就会自动把"明天"转换成具体的日期，把"2月28日"补全为标准格式"2026-02-28"。同时IntentionAgent的第3步Query改写会把标准化后的日期输出到rewritten_query字段中，后续的EventCollectionAgent和ItineraryPlanningAgent直接使用这些标准化的日期，就不需要再做时间解析了。
这种设计让用户可以自然地使用口语化的时间表达，系统能够准确理解并转换为标准格式。

25. 你的Skill Plugins架构和MCP有什么区别
1. 本质不同
- Agent Skills：Agent具备的具体执行能力/工具（如搜索、计算、查数据库、调用API）。
- MCP：一套标准化通信协议，定义模型/Agent如何与外部工具、环境、数据交互。
2. 层级不同
- Skills：功能层，是"能做什么"。
- MCP：交互层/协议层，是"怎么规范地调用"。
3. 作用不同
- Skills：提供实际业务能力，完成具体任务。
- MCP：统一交互格式，让不同Agent、工具、服务能安全、标准化、跨平台互通。
4. 关系 MCP是调用规则，Agent Skills是被规则管理和调用的能力集合。

26. 你的系统如何保证高并发性能
当前版本是CLI原型系统，生产环境规划中主要通过五个方面来保证高并发性能。
第一是异步架构，整个系统基于异步机制实现，所有Agent都采用异步方法（async/await），这样在等待LLM响应时可以处理其他请求，充分利用IO等待时间。
第二是并行Agent调度，通过asyncio.gather让同优先级的Agent同时执行，比如5个信息收集类的Agent可以并行调用，大大提升了吞吐量。
第三是Redis缓存层（生产环境规划），缓存短期记忆、用户偏好热数据和LLM总结结果，减少数据库查询压力。
第四是FastAPI框架（生产环境规划），基于Starlette的异步框架，天生支持高并发。
第五是懒加载机制，Agent只在首次访问时才加载，系统启动时只扫描不加载，减少了冷启动开销。

27. 未来优化（必准备）
结合项目实践经验，我认为主要有以下几个大方向的改进空间。
- 流式响应和思维链可视化：现在用户提问后要等所有Agent执行完才能看到结果，用户体验不好。未来可以实现流式输出，Agent执行完一个就立即返回部分结果，同时把Agent的推理过程可视化展示出来，让用户看到系统正在"思考"什么、调用了哪些Agent、每步得到了什么结果，既提升了用户体验，也增强了系统的可解释性。
- 规则引擎+LLM混合意图识别：目前所有用户输入都要调用LLM来识别意图，响应时间长且成本高。未来可以增加一个规则引擎，对于那些明确的高频意图比如"我要去北京""查天气"，直接用规则匹配秒级响应，不需要调用LLM。只有那些复杂的、模糊的意图才走LLM深度推理。这样既保证了准确率，又大幅提升了响应速度，降低了成本。
- 完善全链路可观测性：现在如果系统出问题了，很难快速定位是哪个Agent哪个环节出了问题。未来可以建立分布式追踪体系，记录每个Agent的调用链路、执行时间、token消耗，从Trace级别追踪单次请求，从Session级别分析用户会话。
- 数据库升级：当前版本使用JSON文件和内存做存储，生产环境需要引入PostgreSQL做持久化存储、Redis做缓存层，提升数据安全性、查询性能和并发能力。
- Milvus Lite升级到Milvus服务器版：当前版本使用Milvus Lite单机部署，如果数据量和并发量上去后，可以迁移到Milvus服务器版（分布式架构，支持水平扩展），同时引入BM25混合检索进一步提升召回准确率。

29. 这个项目是用Vibe Coding的方式开发的，能说一下你的开发思路吗（开放题，请结合自己对vibe coding的理解说，以下仅作参考）
注意：不同面试官对vibe coding的态度不同，只在对方先提到的时候我们才来谈论vibe coding
对，这个项目主要是借助AI编程工具来辅助开发的，但我想强调的是，Vibe Coding不是让AI随便生成一堆代码然后就用了，核心还是人来主导架构设计和技术决策，AI负责加速实现。 整体开发思路为：
1. 架构设计由人主导。在动手写代码之前，我先花了比较多的时间做架构规划。比如设计两层记忆系统的分层策略，确定插件化的Skill Plugins架构方向，这些核心决策都是基于我对业务需求和技术方案的理解做出的。AI工具在这个阶段的作用主要是帮我快速调研技术方案、对比不同框架的优劣，比如AgentScope和LangGraph的对比，但最终拍板是我自己。
2. 模块实现用AI加速。确定好架构后，每个模块的实现我会先写清楚需求描述和接口规范，然后让AI生成初版代码，我再review和调整。举个例子，IntentionAgent的Prompt设计、LazyAgentRegistry的动态加载机制、调度器的优先级并行调度逻辑，这些我都是先在脑子里想清楚核心流程，用自然语言描述给AI，让它生成代码框架，然后我逐行审查。特别是Prompt工程的部分，AI生成的初版往往需要经过好几轮的调试和优化才能达到预期效果。
3. 迭代优化靠实测驱动。AI生成的代码跑起来之后，很多问题是在实测中发现的。比如意图识别准确率最初只有65%，我通过收集1000条测试集反复验证，逐步调整Prompt和意图定义，才把准确率提到90%以上。再比如串行调度导致响应时间30秒太慢，我设计了优先级并行调度方案，让AI帮我实现asyncio并行的代码，然后实测验证确实降到了15秒。这个过程中AI是执行者，但发现问题、定义优化方向、设计解决方案都是我来做的。
我对Vibe Coding的理解是：它本质上是把开发者的角色从"写代码的人"转变成"架构师+产品经理+代码审查者"。你需要有足够的技术判断力来做架构决策，有清晰的需求表达能力来指导AI，还要有严格的代码审查意识来保证质量。如果只是让AI随意生成代码而不加审查，产出的系统质量是不可控的。

29.1 在AI辅助编程的过程中, 你具体是怎么给AI下指令
我总结出一个原则:越具体越好,越结构化越好。拿意图识别模块举例,我不会说"帮我写一个意图识别Agent",而是会给AI一份详细的需求描述,包括这个Agent要接收什么格式的输入、核心处理逻辑分几步、输出的JSON结构长什么样、要考虑哪些边界情况。比如我会明确告诉AI:"这个Agent需要支持多意图识别,每个意图要有置信度;需要对口语化的用户输入进行Query改写;需要输出一个包含推理过程、意图列表、改写后的查询、以及Agent调度计划的JSON"。另外我会在指令中附带示例,比如"当用户输入'请规划一下周末杭州两日游'时,期望的输出是什么样的",这样AI对预期结果就很清晰了。对于Prompt工程的部分更是如此,我会先用自然语言写好Prompt的骨架,包括角色设定、任务步骤、输出格式要求、重要提示和约束条件,然后让AI帮我细化和补充,再反复调试效果。

29.2 怎么拆解需求
我的思路是自顶向下拆分,每次只给AI一个模块的任务。整个多Agent系统很复杂,如果一次性让AI写完,代码质量肯定不行。所以我会先画好整体架构:哪些模块、模块之间怎么通信、数据流是什么样的。然后按模块逐个推进,比如先实现IntentionAgent的意图识别,稳定后再做OrchestrationAgent的调度逻辑,再做各个子Agent。每个模块我还会继续拆分,比如实现插件化架构时,我会先让AI实现"扫描目录发现Skill"的功能,验证可用后,再实现"动态导入模块并实例化Agent"的功能,最后再实现"缓存和懒加载"的优化。这种增量式的开发方式,每一步都能验证正确性,不会出现"一堆代码全是Bug"的情况。

29.3 怎么审查代码
AI生成的代码我从来不会直接用,必须经过系统性的审查。我的审查流程分三步走:
第一步是"先看骨架,再看细节"。拿到AI生成的代码,我不会马上看每一行怎么写的,而是先看整体结构对不对——模块划分是否清晰、类和函数的职责是否单一、模块之间的依赖关系是否合理。如果骨架就有问题,细节写得再好也没用,这时候我会直接打回去重新生成,而不是在错误的架构上修修补补。
第二步是"对照接口契约逐项检查"。因为我在需求拆解阶段就定义好了每个模块的输入输出规范,所以审查时就对照这些契约来检查:输入格式是否正确处理了、输出结构是否符合约定、异常情况是否有兜底策略、边界条件是否覆盖到了。AI生成的代码有个通病——正常流程写得很漂亮,但异常处理几乎为零。比如调用大模型可能返回非标准格式,网络请求可能超时,并行执行时某个子任务可能失败,这些AI不会主动考虑,需要我来补充完善。
第三步是"从使用者视角反向验证"。我会假设自己是下游模块的开发者,看这个模块的接口是否好用、返回值是否容易理解、错误信息是否有助于排查问题。这一步经常能发现AI遗漏的设计问题,比如返回的错误信息不够具体、缺少必要的日志、模块间的命名约定不统一等。
总的来说,代码审查的核心原则是:架构决策必须由人来把关,AI只负责实现细节。AI擅长写符合语法的代码,但对系统级别的设计权衡、模块间的协作逻辑、生产环境的鲁棒性要求,缺乏全局判断力,这些必须由开发者自己把控。

29.4 怎么调试Bug
我的调试方法是分层验证:
首先是单模块测试,每开发完一个Agent就单独写测试脚本跑一遍。比如意图识别模块的测试脚本,我准备了多种类型的测试用例——行程规划、偏好收集、信息查询等,验证每种意图能否正确识别。
然后是集成测试,把多个Agent串起来测试整个流程。这时候经常发现模块间通信的问题,比如IntentionAgent输出的调度计划中Agent名称和注册表里的名称不一致,导致调度器找不到对应的Agent。这种Bug只有在集成测试时才会暴露。
第三是用真实数据测试,我收集了1000条真实用户输入构建测试集,批量跑意图识别,统计准确率。最初只有65%,我通过分析错误case和对项目的逐步优化,最终把准确率提到90%以上。这个过程完全靠人来分析和决策,AI只是帮我快速修改代码。
最后是日志和可观测,我在关键节点都加了日志,比如意图识别的输入输出、Agent调度计划、每个Agent的执行结果。当线上出现问题时,可以通过日志快速定位是意图识别错了还是某个Agent执行失败了。

总结一下,Vibe Coding的本质不是"让AI写代码你就不用管了",而是要把自己变成一个高效的技术指挥官——你负责战略(架构设计、需求拆解),AI负责战术执行(代码生成),你再做最终的质量把关(代码审查、调试验证)。整个过程中,人的技术判断力是不可替代的。

30. 用户问无关问题怎么办
无关问题在意图识别的时候会被IntentionAgent识别为闲聊意图。我在Prompt里定义了闲聊的识别规则，当识别为闲聊时，系统不调用任何Skill，直接返回预设回复，礼貌告知"我主要提供差旅出行与行程规划相关的服务"，并引导用户提出相关需求。这样既不会浪费token调用无用的Agent，也能给用户一个明确的边界感。
