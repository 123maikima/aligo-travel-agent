## 执行版迭代计划

## Phase 1: 安全加固（最高优先级）

### 1.1 JWT Secret 强制校验

**目标**：启动时检测 `API_JWT_SECRET` 是否为默认值，阻止服务启动。

**改动文件**：
- `travel_agent/config.py` — 新增 `validate_secrets()` 函数
- `travel_agent/web_api.py` — `create_app()` 中调用校验
- `travel_agent/cli.py` — CLI 启动入口调用校验

**实现要点**：
- 校验 `API_JWT_SECRET`、`LLM_API_KEY`、`POSTGRES_PASSWORD`（若启用）
- 环境变量为 `"replace-me"`、`"changeme"`、空字符串时抛出 `RuntimeError`
- 提供 `ALLOW_INSECURE_STARTUP=true` 环境变量绕过（仅开发用，启动日志打印 WARNING）
- 不要硬编码死密码列表，改为检测 `len(secret) < 16` 作为最低长度要求

**注意事项**：
- 不要修改现有 `.env` 或 `.env.example`，只改代码逻辑
- 错误信息不要打印密钥本身
- `ALLOW_INSECURE_STARTUP` 仅用于 CI 和本地开发，文档中需注明风险

### 1.2 API 限流

**目标**：防止单用户刷爆 LLM 配额。

**改动文件**：
- `requirements.txt` — 新增 `slowapi`
- `travel_agent/web_api.py` — 接入 `SlowAPI` 中间件
- `travel_agent/config.py` — 新增 `RATE_LIMIT_*` 配置项

**实现要点**：
- 默认限制：`/api/v1/chat` 60 req/min per IP，`/api/v1/auth/token` 10 req/min per IP
- 使用 `SlowAPI`（基于 redis 或内存存储）
- 限流 key 优先用 `user_id`（有 auth 时），fallback 到客户端 IP
- 返回 `429 Too Many Requests` + `Retry-After` header
- 限流配置从环境变量读取：`RATE_LIMIT_CHAT=60/minute`、`RATE_LIMIT_TOKEN=10/minute`

**注意事项**：
- 内存存储模式下多容器不共享限流计数，需在文档注明
- Redis 模式下 Redis 不可用时应降级为内存模式而非 500
- 不要在限流逻辑中调用 LLM 或数据库

### 1.3 消息体长度限制

**目标**：防止超大请求消耗 LLM token 预算。

**改动文件**：
- `travel_agent/web_api.py` — `ChatRequest` Pydantic model
- `travel_agent/config.py` — 新增 `MAX_MESSAGE_LENGTH` 配置

**实现要点**：
- `message: str = Field(..., min_length=1, max_length=4096)`
- `config.py` 中 `MAX_MESSAGE_LENGTH = int(os.getenv("MAX_MESSAGE_LENGTH", "4096"))`
- CLI 端也做同样限制（`cli.py` 输入入口）

**注意事项**：
- 4096 字符约 1500-2000 token，是合理上限
- 超限返回 400 而非截断，让用户知道被拒了

### 1.4 可观测性 PII 脱敏

**目标**：日志中不记录个人行程信息。

**改动文件**：
- `travel_agent/observability.py`

**实现要点**：
- `TraceContext.emit` 中的 `query` 字段做哈希或截断（取前 64 字符 + SHA256 hash）
- 新增 `OBSERVABILITY_MASK_PII=true` 环境变量开关（默认开启）
- 不脱敏 `user_id` 和 `session_id`（这些是追踪必需）
- `error` 字段过滤 URL 中的 API key（正则 `\b[A-Za-z0-9_-]{20,}\b` 替换为 `***`）

**注意事项**：
- 脱敏是单向的，hashed 后不可逆，调试时需配合 `OBSERVABILITY_MASK_PII=false`
- 不要在脱敏函数中引入 LLM 调用（性能敏感路径）

## Phase 3: 架构优化

**改动文件**：

- `travel_agent/cli.py:35-37` — 改为从 registry获取

**实现要点**：

**注意事项**：
- 不要改变 IntentionAgent 的初始化参数（memory_manager, model）
- 改动后确认所有测试通过，因为很多测试直接构造 agent

### 3.2 PostgreSQL 连接池

**目标**：避免每次查询新建连接。

**改动文件**：
- `travel_agent/context/postgres_storage.py` — 引入连接池
- `travel_agent/config.py` — 新增 `POSTGRES_POOL_SIZE`、`POSTGRES_MAX_OVERFLOW` 配置

**实现方案选择**（二选一）：

**方案 A：psycopg2 连接池（推荐，改动最小）**
- 使用 `psycopg2.pool.SimpleConnectionPool` 或 `psycopg2.pool.ThreadedConnectionPool`
- `PostgresLongTermStore` 构造时创建 pool，`close()` 时释放
- 改动集中在 `_connect()` 方法

**方案 B：asyncpg（异步，改动大但性能更好）**
- 需要将所有 `execute` 调用改为 async
- `PostgresLongTermStore` 整个类改 async
- 好处是与其他 async 组件一致

**推荐方案 A**，理由：
- 当前代码是同步的，改为 async 是另一个独立任务
- `psycopg2.pool` 改动量小，风险低
- 性能提升已足够（连接复用 vs 每次新建）

**实现要点**：
- pool 大小默认 `min=2, max=10`，从环境变量可配
- `_connect()` 从 pool 获取连接，返回后自动归还
- 新增 `close()` 方法关闭 pool
- 应用关闭时调用 `close()`（web API 的 shutdown event）

**注意事项**：
- `psycopg2.pool` 在连接耗尽时会抛 `PoolError`，需要捕获并返回有意义的错误
- 确保连接归还（用 try/finally）
- 不要改变 SQL 查询逻辑

### 3.3 SessionStore 持久化

**目标**：多容器部署时 session 状态共享。

**改动文件**：
- `travel_agent/services/session_store.py` — 新增 Redis-backed 实现
- `travel_agent/config.py` — 新增 `SESSION_STORE_BACKEND=memory|redis`

**实现要点**：
- 定义 `SessionBackend` 协议/抽象基类：`register`, `revoke`, `is_revoked`, `get`, `list_user_sessions`, `touch`
- `MemorySessionBackend`（现有实现）
- `RedisSessionBackend`（新增）：用 Redis hash 存 session，TTL 跟随 JWT TTL
- `SessionStore` 构造时根据配置选择 backend
- Redis key 格式：`session:{user_id}:{token}`

**注意事项**：
- Redis 不可用时自动 fallback 到 memory backend
- 不要改变 `SessionStore` 的 public API
- Redis 连接复用已有的 `redis_cache.py` 中的连接

### 3.4 重试与降级机制

**目标**：为 PostgreSQL、Milvus、LLM API 添加指数退避重试。

**改动文件**：
- `travel_agent/utils/resilience.py` — 已有 `retry_with_backoff`，扩展使用范围
- `travel_agent/context/postgres_storage.py` — `save_snapshot`、`load_snapshot` 包裹重试
- `travel_agent/agents/rag_knowledge_agent.py` — `_ensure_connection` 改为多次重试
- `travel_agent/context/memory_manager.py` — 长时摘要生成包裹重试

**实现要点**：
- 扩展 `retry_with_backoff` 支持 async 函数（如果尚未支持）
- 默认 `max_retries=3, base_delay=1.0, max_delay=30.0`
- 仅对可重试错误重试（网络超时、连接拒绝），不对业务错误重试
- `is_retriable_error` 函数判断错误类型

**注意事项**：
- LLM API 调用已有 `circuit_breaker.py`，不再额外加重试（circuit breaker 本身就是降级机制）
- PostgreSQL 重试仅针对连接级错误（OperationalError），不针对 SQL 错误
- Milvus 重试仅针对 gRPC 连接错误

### 4.2 Docker 多阶段构建

**目标**：减小镜像体积，移除 build 工具。

**改动文件**：
- `Dockerfile` — 改为多阶段构建
- `docker-compose.yml` — 统一 app/web 服务

**Dockerfile 实现要点**：

```dockerfile
# Stage 1: builder
FROM python:3.11-slim AS builder
RUN apt-get update && apt-get install -y --no-install-recommends build-essential git
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

# Stage 2: runtime
FROM python:3.11-slim
WORKDIR /app
COPY --from=builder /install /usr/local
COPY . .
RUN useradd -m appuser && chown -R appuser:appuser /app
USER appuser
EXPOSE 8000
CMD ["uvicorn", "travel_agent.web_api:app", "--host", "0.0.0.0", "--port", "8000"]
```

**docker-compose.yml 实现要点**：
- `app` 和 `web` 用同一 image，仅 `command` 不同
- 新增 `healthcheck` 对 web 服务（curl `/api/v1/health`）
- 移除 build-time 的 `POSTGRES_PASSWORD` 默认值

**注意事项**：
- `python:3.11-slim` 体积小但需确认所有 native 依赖能编译
- 如果有 `psycopg2`，考虑换成 `psycopg2-binary` 或在 builder 阶段编译
- 健康检查路径需要 web API 支持 `/api/v1/health` 端点

### 4.3 日志轮转

### 0.1 定义项目级 Agent Protocol / ABC

**现状问题**：所有 agent 继承 `agentscope.agent.AgentBase`，但项目自身没有定义接口契约。`__init__` 签名各异，`reply` 参数风格不一。Orchestrator 通过 `AgentRegistry(Protocol)` 约束了注册表，但对 agent 本身没有任何约束。

**目标**：定义 `travel_agent/agents/base.py`，提供明确的接口契约。

**改动文件**：
- 新增 `travel_agent/agents/base.py`
- 所有 agent 改为继承 `BaseTravelAgent` 而非直接继承 `AgentBase`

**实现要点**：

```python
class AgentContext(TypedDict, total=False):
    """Agent 接收的结构化上下文，替代当前 JSON-serialized dict"""
    rewritten_query: str
    intents: list[dict]
    key_entities: dict
    recent_dialogue: list[dict]
    user_preferences: dict
    previous_results: list[dict]
    reason: str
    expected_output: str

class BaseTravelAgent(AgentBase):
    """所有 travel agent 的基类。

    子类必须实现:
        async def _execute(self, ctx: AgentContext) -> dict

    基类负责:
        - 解析 Msg 输入 → AgentContext
        - 将 _execute 返回值包装为 Msg
        - 统一错误处理和超时日志
        - 统一 LLM 响应文本提取
    """
    async def reply(self, x: Msg | list[Msg] | None = None) -> Msg: ...
```

**注意事项**：
- 不要删除 `AgentBase` 继承，而是在其上加一层项目级抽象
- `AgentContext` 是强类型 dict，替代当前各 agent 各自 `json.loads(msg.content)` 的解析逻辑
- `_execute` 返回纯 `dict`（当前做法），由基类包装为 `Msg(content=json.dumps(result))`
- 每个 agent 改动需独立 commit，逐步迁移

**收益**：
- 新增 agent 时 IDE 自动补全 `AgentContext` 字段
- 基类统一处理 JSON 序列化/反序列化，消除各 agent 中的重复解析
- 测试时可 mock `_execute` 而无需构造完整 Msg

### 0.3 编排层 Agent 优先级动态化

**现状问题**：`OrchestrationAgent` 中的优先级分组（P1 并行、P2 串行）是硬编码的（line 182-208）。新增 agent 必须修改 `_run_agents()` 中的分支逻辑。

**目标**：优先级从 `SKILL.md` 的 `intent_priority` 读取，编排逻辑完全通用。

**改动文件**：
- `travel_agent/agents/orchestration_agent.py` — 重构 `_run_agents()`

**当前代码结构**（简化）：
```python
# P1 并行
p1_agents = [name for name in agent_names if name in P1_SET]
# P2 依赖 P1
p2_agents = [name for name in agent_names if name in P2_SET]
```

**重构为**：
```python
from collections import defaultdict

def _group_by_priority(self, agent_names: list[str], schedule: list[dict]) -> list[list[str]]:
    """返回按优先级分组的 agent name 列表。"""
    groups: dict[int, list[str]] = defaultdict(list)
    for name in agent_names:
        priority = self._resolve_priority(name)  # 从 SKILL.md 或 fallback
        groups[priority].append(name)
    return [groups[k] for k in sorted(groups)]
```

**实现要点**：
- `_resolve_priority(name)` 优先从 `SkillLoader` 读取 `intent_priority`，fallback 到默认值 `1`
- 同优先级内自动并行（`asyncio.gather`），不同优先级按数字顺序串行
- 删除硬编码的 `P1_SET` / `P2_SET`
- `AGENT_PRIORITY_MAP` 保留作为紧急 fallback，但标注 deprecated

**注意事项**：
- 优先级数字越小越先执行（P0 > P1 > P2），与当前一致
- `itinerary_planning` 依赖 `event_collection` 的结果，这个依赖关系通过优先级（`event_collection: 1`, `itinerary_planning: 2`）隐式表达
- 如果未来出现复杂依赖（如 agent A 依赖 agent B 和 C，但 B 和 C 优先级不同），需要考虑引入 DAG 调度而非简单优先级。当前架构不需要

### 0.4 Agent 间通信协议强化

**现状问题**：
- 各 agent 通过 `json.dumps(dict)` 通信，但接收方各自解析，处理方式不一
- `previous_results` 中查找上游结果靠字符串匹配 `agent_name == "event_collection"`（`itinerary_planning_agent.py:98-102`），极易断裂
- 没有统一的错误结果 schema，各 agent 的 error 返回格式不同

**目标**：定义标准化的 `AgentResult` 数据结构 + 统一的通信序列化。

**改动文件**：
- 新增 `travel_agent/agents/protocol.py`
- 所有 agent 改用 `AgentResult` 返回

**实现要点**：

```python
@dataclass
class AgentResult:
    """Agent 执行结果的标准结构。"""
    agent_name: str       # 自动填充
    status: Literal["success", "error", "skipped"]
    data: dict            # 业务数据（行程、事件、偏好等）
    error: str | None = None
    metadata: dict = field(default_factory=dict)  # token 用量、耗时等

    def to_msg(self, sender: str = "Agent") -> Msg: ...
    @classmethod
    def from_msg(cls, msg: Msg) -> "AgentResult": ...
    @classmethod
    def from_error(cls, agent_name: str, error: str) -> "AgentResult": ...
```

**统一错误格式**（替代当前各 agent 各自返回 `{"error": "..."}` 的做法）：
```python
# 成功
AgentResult(agent_name="event_collection", status="success", data={"events": [...], "extracted_count": 3})

# 失败
AgentResult(agent_name="event_collection", status="error", data={}, error="LLM timeout", metadata={"retry_count": 0})
```

**`previous_results` 查找改为类型安全**：
```python
# 之前（脆弱）
for r in previous_results:
    if r.get("agent_name") == "event_collection":
        ...

# 之后（类型安全）
events_result = next((r for r in previous_results if r.agent_name == "event_collection"), None)
```

**注意事项**：
- `AgentResult.to_msg()` / `from_msg()` 内部处理 `json.dumps` / `json.loads`，各 agent 不再直接调用
- `AgentBase._execute()` 基类方法（见 0.1）自动将返回值包装为 `AgentResult`
- 向后兼容：`from_msg` 能解析旧格式的 dict（各 agent 逐步迁移）
- `metadata` 字段为可观测性提供便利（token 计数、耗时、重试次数等）

### 0.5 意图意图分类器改进

**现状问题**：`IntentionAgent` 使用 LLM + 规则 fallback 做意图分类。LLM 路径走完整的 prompt 构建 → API 调用 → JSON 解析流程，开销大。规则 fallback（`RULE_PATTERNS`）是兜底但覆盖率低。

**目标**：引入轻量级意图预分类，减少不必要的 LLM 调用。

**改动文件**：
- `travel_agent/agents/intention_agent.py` — 新增两级分类流水线

**实现方案**：

```
用户输入 → [预分类器] → 高置信度? → 直接返回结果
                      ↓ 低置信度
                    [LLM 分类器] → 返回结果
```

**预分类器实现**（规则 + 关键词匹配，可选后续接 embedding 相似度）：
- 基于 `intent_patterns` 的正则匹配（当前 fallback 逻辑提升为主路径）
- 匹配成功返回 `(intent_type, confidence=0.8)`
- 无匹配返回 `(None, confidence=0.0)`，走 LLM

**LLM 分类器改进**：
- 预分类无结果时才构建完整 prompt
- LLM 返回结果与预分类结果可以做一致性校验（可选）
- 缓存高频查询的意图结果（已有 `_skill_prompt_cache`，可扩展为 `_intent_cache`）

**注意事项**：
- 不要删除 LLM 分类器，它是高质量分类的保障
- 预分类器的 regex 来自 `SKILL.md` 的 `intent_patterns`（见 0.2）
- 置信度阈值 `0.8` 从 config 可配：`INTENT_RULE_CONFIDENCE_THRESHOLD`
- 监控预分类命中率，如果 < 30% 说明规则覆盖不足，需要补充 patterns
- 这是性能优化，不改变外部行为

### 0.6 Agent 可测试性改进

**现状问题**：
- Agent 直接调用 `self.model(messages)`，无法在不打网络的情况下测试
- 没有依赖注入接口
- 16 个测试文件中仅 5 个是纯单元测试

**目标**：让所有 agent 支持 mock model 运行。

**改动文件**：
- `travel_agent/agents/base.py` — model 通过依赖注入
- 新增 `tests/mocks.py` — `FakeModel` 类

**实现要点**：

```python
class FakeModel:
    """用于单元测试的假 LLM model。"""
    def __init__(self, responses: list[str] | Exception):
        self._responses = responses
        self._call_count = 0

    def __call__(self, messages, **kwargs):
        resp = self._responses[self._call_count % len(self._responses)]
        self._call_count += 1
        if isinstance(resp, Exception):
            raise resp
        return FakeResponse(resp)

class FakeResponse:
    text: str
    def __init__(self, text): self.text = text
```

**使用方式**：
```python
def test_event_collection_extracts_city():
    agent = EventCollectionAgent(
        name="test",
        model=FakeModel(['{"events": [{"type": "出发城市", "value": "上海"}]}'])
    )
    result = asyncio.run(agent.reply(Msg("user", "我从上海出发", "user")))
    data = json.loads(result.content)
    assert any(e["value"] == "上海" for e in data["events"])
```

**各 agent 改造**：
- `__init__` 中 `self.model = model` 保持不变（AgentScope 约定）
- 但 `LazyAgentRegistry` 实例化时传入的 model 可从 config 切换
- 测试中直接构造 agent 并传入 `FakeModel`
- `InformationQueryAgent`（无 LLM 调用）和 `PreferenceAgent`（纯规则）最易改

**注意事项**：
- `FakeModel` 需要支持同步和 async 两种调用模式（AgentScope 的 model 可能返回 async generator）
- 对于 async generator 返回的 model，需要 `FakeAsyncModel` 变体
- 不要改变生产代码中的 model 初始化逻辑，只在测试中使用 `FakeModel`

### 0.8 Orchestrator 重试与降级策略

**现状问题**：
- Orchestrator 对失败的 agent **不重试**（仅超时保护）
- 没有降级策略（如 agent 失败时使用缓存结果或跳过）
- `return_exceptions=True` 在 `asyncio.gather` 中吞掉了并行失败

**目标**：为关键 agent 添加重试 + 降级。

**改动文件**：
- `travel_agent/agents/orchestration_agent.py` — `_run_agents()` 添加重试
- `travel_agent/agents/orchestration_agent.py` — 新增降级策略

**实现要点**：

**重试**（利用已有的 `retry_with_backoff`）：
```python
async def _run_single_agent(self, agent_name: str, context: dict) -> AgentResult:
    agent = self.agent_registry[agent_name]
    return await retry_with_backoff(
        self._execute_agent, agent, context,
        max_retries=2, is_retriable=is_llm_retriable
    )
```

**降级策略**：
- `event_collection` 失败 → 使用空事件列表，标记 `extracted_count=0`
- `memory_query` 失败 → 返回空偏好/历史列表
- `rag_knowledge` 失败 → 返回原始检索文本（无 LLM 摘要）
- `itinerary_planning` 失败 → 返回空模板 + 错误提示
- `preference` 失败 → 静默跳过（不影响其他 agent）

**注意事项**：
- 重试仅针对 LLM 超时/网络错误，不针对 JSON 解析失败（说明 prompt 或 model 输出有问题，重试无益）
- 降级策略的配置应从 config 可配，便于紧急关闭
- 降级后返回的 `AgentResult` 中 `metadata.downgraded = True`，便于观测
- 不重试 `PreferenceAgent`（纯规则，失败不是网络问题）和 `InformationQueryAgent`（外部 API 失败重试可能加重问题）

### 0.9 AgentScope 深度集成评估

**现状问题**：
- AgentScope 仅作为 `AgentBase` + `Msg` 的薄封装使用
- AgentScope 自带的 tool-use、multi-agent pipeline、memory 等能力未使用
- 项目自建了完整的 orchestration、memory、LLM SDK facade，与 AgentScope 生态脱节

**目标**：评估是否应该深入集成 AgentScope，或维持当前自研架构。

**评估维度**：

| 维度 | 自研（当前） | AgentScope 生态 | 建议 |
|------|-------------|----------------|------|
| 编排引擎 | `OrchestrationAgent` 自定义 | `agentscope.pipeline` | 维持自研（更贴合业务） |
| Tool Use | 无 | AgentScope 内建 tool framework | 考虑引入（适合 weather/query 类 agent） |
| Memory | 自建 `MemoryManager` | `agentscope.memory` | 维持自建（已深度定制） |
| LLM SDK | 自建 `UnifiedChatModel` | `OpenAIChatModel` | 维持自建（已支持多 provider） |
| Agent 基类 | `AgentBase` | `AgentBase` | 不变 |
| 消息类型 | `Msg` | `Msg` | 不变 |

### 0.10 用户意图冲突检测与消解

**现状问题**：
- `IntentionAgent` 的 `intents` 是列表（`intention_agent.py:217-223`），LLM 可输出多意图，系统**不检查冲突**直接全部调度。
- `OrchestrationAgent` 同一优先级意图全部并行执行（`orchestration_agent.py:276-278`），矛盾意图同时执行。
- `MemoryUpdater` 直接应用 preference agent 结果到长期记忆，不校验与已有偏好的矛盾。
- 典型冲突场景：
  - `"我想去北京，算了不去了"` — 即时否定，自相矛盾
  - `"上次去北京出差，这次想去北京旅游"` — 同目的地不同目的
  - `"我不喜欢汉庭了，帮我规划去上海住汉庭附近"` — 否定偏好 + 隐式恢复
  - `"我预算有限，但想住万豪"` — 预算与酒店档次矛盾
  - 出发地和目的地相同

**目标**：在意图调度前检测冲突，提供消解策略。

**改动文件**：
- 新增 `travel_agent/agents/intent_conflict_resolver.py`
- `travel_agent/agents/orchestration_agent.py` — 调度前调用冲突检测

```python
class ConflictResolution:
    conflict_type: ConflictType
    severity: Literal["error", "warning", "info"]
    description: str
    action: Literal["drop_low_confidence", "merge", "ask_user", "pass"]
    affected_intents: list[str]
    resolved_intents: list[dict]

class IntentConflictResolver:
    def resolve(self, intention_data: dict, user_preferences: dict) -> list[ConflictResolution]:
        """输入 IntentionAgent 输出 + 用户偏好，返回冲突消解报告。"""
        resolutions = []
        intents = intention_data.get("intents", [])
        entities = intention_data.get("key_entities", {})
        schedule = intention_data.get("agent_schedule", [])

        resolutions.extend(self._check_intent_conflicts(intents, schedule))
        resolutions.extend(self._check_entity_conflicts(entities))
        resolutions.extend(self._check_preference_conflicts(entities, user_preferences))
        resolutions.extend(self._check_immediate_negation(
            intention_data.get("rewritten_query", "")))

        # 修正 agent_schedule
        intention_data["agent_schedule"] = self._apply_resolutions(schedule, resolutions)
        return resolutions
```

**配置化消解策略**：

| `INTENT_CONFLICT_POLICY` | 行为 | 适用场景 |
|--------------------------|------|---------|
| `warn`（默认） | 记录警告，自动消解，不阻断执行 | 生产环境 |
| `resolve` | 自动消解所有冲突，不输出警告 | 高度自动化 |
| `reject` | error 级冲突直接拒绝，要求用户澄清 | 严格模式 |

**注意事项**：
- 冲突检测必须在意图调度**之前**，不能等 agent 跑完才发现矛盾
- `warn` 模式不阻断执行，只在日志和回复中附加说明
- `reject` 模式仅对 `error` 级冲突生效
- 即时否定检测是**纯规则**的，不需要 LLM，开销可忽略
- 偏好违背检测依赖已有偏好的完整性 — 未设置过偏好的用户不会触发
- 冲突消解器应该是**无状态纯函数**，方便单元测试
- 不要试图检测所有可能的矛盾 — 只做确定性高、价值大的检测。模糊矛盾交给下游 agent 处理
- 如果消解后 `agent_schedule` 为空，返回友好的澄清回复而非空结果

### 0-B.2 JSON 并发写入安全

**现状问题**：
- `_save()`（`long_term_memory.py:176`）用 `open(db_path, 'w')` 直接写文件，无文件锁。
- 两个并发请求同时写同一用户的 JSON → torn write → 文件损坏。
- `data/memory/{user_id}.json` 被损坏后，下次 `_load()` 会走到 `except` 分支，返回空数据，**用户数据丢失**。

**目标**：写入加锁，确保原子性。

**改动文件**：
- `travel_agent/context/long_term_memory.py` — `_save()` 加文件锁 + 原子写入

**实现方案**：

```python
import fcntl

def _save(self):
    # 1. 写临时文件
    tmp_path = self.db_path + ".tmp"
    self.data["updated_at"] = datetime.now().isoformat()
    content = json.dumps(self.data, ensure_ascii=False, indent=2)

    # 2. 加排他锁写入
    with open(self.db_path, 'r+', encoding='utf-8') as f:
        fcntl.flock(f, fcntl.LOCK_EX)
        try:
            f.seek(0)
            f.write(content)
            f.truncate()
        finally:
            fcntl.flock(f, fcntl.LOCK_UN)
```

**更安全的原子写入方案**（推荐）：
```python
def _save(self):
    tmp_path = self.db_path + ".tmp"
    self.data["updated_at"] = datetime.now().isoformat()
    with open(tmp_path, 'w', encoding='utf-8') as f:
        json.dump(self.data, f, ensure_ascii=False, indent=2)
        f.flush()
        os.fsync(f.fileno())
    os.replace(tmp_path, self.db_path)  # 原子替换
```

**注意事项**：
- `os.replace()` 在同一文件系统上是原子操作（rename syscall）
- `os.fsync()` 确保数据刷盘，防止断电丢失
- 如果 PostgreSQL 是唯一后端（JSON 仅作为 fallback），可以用轻量锁或直接跳过 JSON 写入
- Windows 上 `os.replace()` 也支持原子替换，但 `fcntl.flock` 不可用 — 项目跑在 Linux 上，不需要兼容 Windows
- 推荐原子写入方案（临时文件 + rename），比 fcntl 简单且跨平台

```python
def _has_negation(self, query: str, pref_value: str) -> bool:
    """检测 pref_value 是否在否定上下文中。"""
    # 简单方法：检查否定词是否出现在 pref_value 附近
    idx = query.find(pref_value)

    context = query[max(0, idx-6):idx]  # 取前 6 个字符
    return any(neg in context for neg in self.NEGATION_WORDS)
```

**时间衰减策略**：
```python
def get_preference_with_decay(self, pref_type: str, decay_half_life_days: int = 180) -> Any:
    """获取偏好，应用时间衰减。

    如果偏好的 updated_at 距今超过 decay_half_life_days * 2，返回 None（视为过期）。
    """
    pref = self._find_preference(pref_type)
    if not pref:
        return None

    updated = datetime.fromisoformat(pref.get("updated_at", ""))
    age_days = (datetime.now() - updated).days
    half_life = decay_half_life_days

    # 衰减系数：exp(-ln(2) * age / half_life)
    decay = math.exp(-math.log(2) * age_days / half_life)
    confidence = pref.get("confidence", 0.8) * decay

    if confidence < 0.2:  # 阈值以下视为无效
        return None

    return pref["value"]
```

**冲突解决**：
```python
def save_preference(self, pref_type: str, value: Any, **kwargs):
    # 如果新偏好与已有偏好矛盾（value 不同且 sentiment 相反），
    # 不直接覆盖，而是标记旧偏好为 "superseded"
    existing = self._find_preference(pref_type)
    if existing and existing.get("value") != value:
        if existing.get("sentiment") == "positive" and kwargs.get("sentiment") == "negative":
            # 正面 → 负面：标记过期
            existing["superseded_at"] = datetime.now().isoformat()
```

### 0-B.6 Redis 重连接与 STM 异步写入

**现状问题**：
- `RedisCache.__init__`（`redis_cache.py:82-94`）连接失败后 `self.enabled = False`，之后**永不重连**。
- `ShortTermMemory.add_message()`（`short_term_memory.py:82`）**同步**调用 `_save_to_redis`，每个消息增加 Redis RTT 延迟。
- 如果 Redis 在运行中宕机，所有后续缓存操作静默失败但 `enabled` 仍为 `True`（`_set` 捕获异常只 warning，不改 `enabled`）。

**目标**：Redis 自动重连 + STM 写入异步化。

**改动文件**：
- `travel_agent/context/redis_cache.py` — 重连逻辑
- `travel_agent/context/short_term_memory.py` — 异步/批量写入 Redis

**实现要点**：

**重连逻辑**：
```python
class RedisCache:
    RECONNECT_INTERVAL = 30  # 秒
    RECONNECT_MAX_ATTEMPTS = 10

    def __init__(self, ...):
        self._last_reconnect_attempt = 0
        self._reconnect_failures = 0
        ...

    def _ensure_connection(self):
        """懒检查/重连。"""
        if self.enabled and not self._client:
            now = time.time()
            if now - self._last_reconnect_attempt < self.RECONNECT_INTERVAL:
                return  # 冷却中
            self._attempt_reconnect()

    def _attempt_reconnect(self):
        self._last_reconnect_attempt = time.time()
        try:
            self._client = redis.Redis(...)
            self._client.ping()
            self.enabled = True
            self._reconnect_failures = 0
            logger.info("Redis reconnected")
        except Exception:
            self._reconnect_failures += 1
            self._client = None
            if self._reconnect_failures >= self.RECONNECT_MAX_ATTEMPTS:
                self.enabled = False
                logger.warning("Redis reconnect exhausted, cache disabled")
```

在 `_get`, `_set`, `_delete` 开头调用 `self._ensure_connection()`。

**STM 异步批量写入**：
```python
class ShortTermMemory:
    def __init__(self, ...):
        self._dirty = False  # 是否有未同步的变更
        self._flush_interval = 3  # 秒

    def add_message(self, ...):
        self.messages.append(message)
        self._trim_if_needed()
        self._dirty = True
        self._schedule_flush()  # 异步调度，而非同步写入

    def _schedule_flush(self):
        """延迟 flush，合并多次 add_message 为一次 Redis 写入。"""
        if not self._flush_timer:
            self._flush_timer = threading.Timer(self._flush_interval, self._flush_to_redis)
            self._flush_timer.daemon = True
            self._flush_timer.start()

    def _flush_to_redis(self):
        if self._dirty and self._session_id:
            self._save_to_redis(self._session_id)
        self._dirty = False
        self._flush_timer = None
```

**注意事项**：
- 异步 flush 使用 `threading.Timer`（轻量，不需要 asyncio loop）
- `clear()` 和 `set_session()` 时需要立即 flush + reset timer
- 进程退出时需要 flush 剩余未写入的数据（注册 `atexit` 钩子）
- 如果 STM 不需要 Redis（单进程部署），异步写入完全无影响
- 重连间隔 30s 是合理的，不要太频繁（避免 Redis 重启期间疯狂重试）

### 0-B.8 PostgreSQL 迁移路径完善

**现状问题**：
- `scripts/migrate_json_to_postgres.py` 没有备份、没有进度追踪、没有回滚路径。
- 迁移是 "JSON → PostgreSQL 镜像"，但系统仍然双写 JSON，JSON 始终是 canonical source。
- 没有 "PostgreSQL-only" 模式 — 无法真正摆脱 JSON 文件。

**目标**：实现完整的迁移路径：JSON → 双写 → PostgreSQL-only。

**改动文件**：
- `scripts/migrate_json_to_postgres.py` — 完善迁移脚本
- `travel_agent/context/long_term_memory.py` — 支持 read-from-PG-only 模式
- `travel_agent/config.py` — 新增 `POSTGRES_MODE=mirror|primary` 配置

**三阶段迁移**：

**阶段 1：Mirror（当前状态）**
- PostgreSQL 是 JSON 的镜像，双写
- 读取优先 JSON

**阶段 2：Primary（新增）**
- `POSTGRES_MODE=primary` 时：
  - 读取优先 PostgreSQL，JSON 仅作为 backup
  - 写入仍然双写
  - `_load()` 先查 PG，miss 则从 JSON 迁移到 PG
- 此阶段用于验证 PG 数据的完整性和正确性

**阶段 3：PG-Only（最终目标）**
- `POSTGRES_MODE=primary` 且运行一段时间后：
  - 停止 JSON 写入
  - JSON 文件可删除或归档
  - 完全依赖 PostgreSQL

**迁移脚本改进**：
```python
# migrate_json_to_postgres.py 改进
1. 迁移前备份：shutil.copytree("data/memory", "data/memory_backup_{timestamp}")
2. 进度追踪：记录已迁移文件到 .migration_progress.json
3. 断点续传：跳过已迁移的文件
4. 数据校验：迁移后对比 JSON 和 PG 的记录数
5. 回滚命令：--rollback 从 backup 恢复
6. 干跑模式：--dry-run 只报告不迁移
```

**注意事项**：
- 不要急于推进到阶段 3，阶段 2 需要运行一段时间验证
- 备份目录需要定期清理（保留最近 3 个备份）
- 数据校验发现不一致时打印 WARNING 但不中断迁移
- `.migration_progress.json` 格式：`{"migrated_files": ["user1.json", ...], "last_run": "..."}`

```python
    "西安", "广州", "深圳", "重庆", "武汉", "南京", "成都", "杭州",
    "昆明", "厦门", "青岛", "大连", "三亚", "拉萨", "桂林", "张家界",
]

async def generate_city_guide(city: str, model) -> dict:
    prompt = f"""请为以下城市生成详细的旅行攻略，输出 JSON 格式：

城市：{city}
要求包含：景点推荐（名称、类别、时长、价位、描述、周边景点）、酒店推荐、美食推荐、交通信息。

"""
```

## 执行顺序建议

```
Phase 0-A（Agent架构）→ Phase 0-B（记忆系统）→ Phase 1（安全）   → Phase 2（代码质量）→ Phase 3（架构）
        ↓                      ↓                   ↓                              ↓
  基础设计优化            记忆核心修复            可立即部署                       需要回归测试
                          ↓                      ↓
                    0-B.1/0-B.2 优先          Phase 4（DevOps）→ Phase 5（测试）    → Phase 6（择机）
                                                    ↓
                                              Phase 0-C（框架/前端/数据）
```

- Phase 0-A 的 0.1（Protocol）和 0.4（Protocol 数据类）应最先完成，是其他 agent 改动的基础
- **Phase 0-A 的 0.10（意图冲突）是用户体验关键路径** — 用户说自相矛盾的话时不应产生错误输出
- **Phase 0-B 的 0-B.1（chat_history 无界增长）和 0-B.2（并发写入安全）是记忆系统最高优先级**，可能导致数据损坏/丢失
- Phase 0-B 的 0-B.3（偏好自适应）和 0-B.4（冲突解决）相互依赖，一起做
- 0.10（意图冲突）和 0-B.4（偏好冲突）互补但不同：0.10 处理单次请求内的多意图矛盾，0-B.4 处理跨时间的偏好演变 — 可并行推进
- Phase 0-A 和 Phase 0-B 可以并行推进（不同子系统，无冲突）
- **Phase 0-C.1（AgentScope 升级）应尽早做**，因为它影响所有后续代码 — 升到 1.0.21 后再做其他改动
- **Phase 0-C.3（数据生成）可在 Phase 5（测试补全）之前完成**，为测试提供数据
- Phase 0-C.2（前端）完全独立，不影响后端，可随时穿插
- Phase 1 和 Phase 2 可以并行推进（不同文件，无冲突）
- Phase 3 的 3.2（连接池）和 3.3（SessionStore）无依赖，可并行
- Phase 4 的 CI/CD 应在 Phase 5 之前完成（测试需要 CI 运行）
- Phase 6 各项相互独立，可随时穿插

## 全局注意事项

1. **每个改动独立 commit**，方便回滚和 code review
2. **不要在一个 PR 中混改多个 Phase**
3. **Phase 0-B.1（无界增长）和 0-B.2（并发安全）优先合入** — 它们是数据安全和性能的关键路径
4. **Phase 1 的安全改动紧随其后**，即使其他 Phase 未完成
5. **所有配置项通过环境变量暴露**，不硬编码
6. **改代码前先读相关测试**，确保不破坏现有行为
7. **涉及外部依赖的改动**（slowapi、fakeredis 等）先在 `requirements.txt` 中锁定版本
8. **Type 注解改动**使用 `mypy --no-error-summary` 或 `ruff check` 验证
9. **Docker 改动**后跑 `docker compose up --build` 确认服务正常启动
10. **PR 描述写清楚 Why**，不是 What（commit message 已有 What）
11. **保留 `ALLOW_INSECURE_STARTUP` 等开发绕过开关**，但文档中醒目标注风险
12. **记忆系统改动必须做数据迁移** — 任何数据结构变更（如偏好扩展字段）需要向前向后兼容
13. **偏好自适应（0-B.3）不改变现有规则提取逻辑** — 是叠加而非替换，规则路径始终是 fallback
14. **PostgreSQL 迁移（0-B.8）是渐进式** — 不要试图一次性切换到 PG-only，先用 mirror 模式验证
15. **原子写入（0-B.2）的 `os.replace()` 必须确保 tmp 文件和目标文件在同一文件系统** — 不能用 `/tmp` 做临时目录
16. **AgentScope 升级（0-C.1）必须先升到 1.0.21，2.0.0 暂缓** — 2.0.0 的 Msg 重构会破坏所有 8 个 agent
17. **前端拆分（0-C.2.6）不影响后端 API** — 所有新增组件只消费已有 SSE 事件流
18. **城市攻略数据（0-C.3.1）生成后必须人工审核** — LLM 生成的价格、距离等信息可能有误
19. **测试 Query 集（0-C.3.3）用 YAML 格式维护** — 方便非开发者增删测试用例
20. **前端只引入 `react-markdown` 一个新依赖** — 不要引入重量级 UI 框架，保持当前轻量风格
